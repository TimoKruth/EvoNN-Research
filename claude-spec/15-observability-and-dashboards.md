# 15 — Observability And Dashboards

Search telemetry is a product surface. The system must explain what it
searched, what it reused, what it discarded, what motifs emerged, and where
time and memory went. Three surfaces serve this: the static fair-matrix
dashboard (ch. 05), the Observatory web application (this chapter), and
autonomous research loops (autoresearch).

## Surface 1 — Static Fair-Matrix Dashboard

Specified in chapter 05: single HTML + JSON payload per workspace; winner
tables (full-system and projects-only); multi-seed evidence with spread, CIs,
pairwise deltas; per-seed snapshots; lane health; runtime tables; Evidence
Explorer; recent-runs overview; transfer provenance. Regenerated by
`fair-matrix` and `workspace-report`; also buildable over the evidence
registry (`dashboard evidence`).

## Surface 2 — Observatory (Read-Only Web Dashboard)

The ancestral Observatory design is carried forward as the umbrella's
human-friendly exploration UI. It is a **read-only** web app over exported
artifacts — no editing, no run triggering, no auth, no database.

### Stack

FastAPI + Jinja2 server-side templates + Chart.js (CDN) + vanilla JS.
**No build step** — no npm, no webpack. Dark theme; responsive ≥900px with
auto-collapsing sidebar.

### Data model

A filesystem scanner walks configured roots on startup and re-scans every
30 seconds; parses `manifest.json` / `results.json` / campaign and
fair-matrix summaries into dataclasses; caches in memory (artifact volume is
small). Every comparison result the UI needs MUST exist as structured JSON
beside its markdown (comparison JSON, leaderboard JSON) — the UI never
scrapes markdown.

### CLI

```
evonn-compare observatory --roots <workspace-or-registry-paths…> --port 8417
```

### Pages

1. **Dashboard (home)** — stat cards (total runs by system, campaigns,
   benchmarks covered, active systems); latest campaign win-distribution
   chart; recent-runs table.
2. **Campaigns list** — name, tier badge, seeds/budgets, win counts,
   significance (p-value) when available, date; sortable.
3. **Campaign detail** — single scrollable page: header; "what is this
   campaign?" banner; stat-card row; per-benchmark win-rate heatmap (color =
   winner, opacity = consistency); statistical analysis (Wilcoxon p, median
   delta, bootstrap 95% CI, mean delta); budget-scaling chart; search
   telemetry (QD metrics when present); per-seed results table linking to
   matchup detail.
4. **Run detail** — system badge; manifest info (pack, budget, device,
   precision, backend); per-benchmark results table with metric direction;
   architecture summary (per engine: family/layers vs DAG depth/skips vs
   hierarchy depth/reuse vs primitive motifs); artifact links; search
   telemetry.
5. **Run lists** — filterable per system.
6. **Leaderboard** — cross-campaign win rates, mean normalized score
   rankings, budget-scaling trends, statistical confidence summary.

### Explanatory notes (hard requirement)

Every chart and data section carries a visible 1–2 sentence plain-language
info banner (not a tooltip) — win-rate heatmap, Wilcoxon, bootstrap CI,
budget scaling, QD telemetry, parity status, architecture summary, median
delta, parameter count all have canonical texts. The dashboard must be useful
without prior knowledge of the project.

### Non-goals

Real-time in-run streaming; editing or triggering runs; authentication;
database persistence.

## Surface 3 — Autonomous Research Loops (autoresearch-mlx)

A self-contained side project and a pattern the umbrella adopts: fixed-budget
autonomous keep-or-revert experimentation, MLX-native.

Core protocol (Karpathy autoresearch, ported to Apple Silicon):

- one mutable `train.py`; one read-only `prepare.py` (data, tokenizer,
  evaluation ground truth); one metric (`val_bpb`); fixed 5-minute training
  wall-clock; `results.tsv` history (commit, metric, memory, keep/discard/
  crash, description)
- loop: edit → commit → run → read metric → keep (advance branch) or revert
  (`git reset --hard` to last kept commit); dedicated
  `autoresearch/<tag>` branch; baseline established per-hardware, never
  borrowed across machines
- simplicity criterion: weigh complexity cost against improvement; a
  deletion that ties is a win
- hardware-local findings are expected to differ across machines (documented
  Max-vs-Mini divergence) — exactly what fixed-time local loops reveal

Umbrella relevance (Horizon 1 automation loop): EvoNN should support a
repeatable automation loop that pulls latest work, runs bounded
package-relevant smoke tests after each small improvement, runs tiny-budget
trend-detection configs, and records whether small-budget runs improve or
regress over time — the compare/report surfaces accumulate these instead of
treating every improvement as anecdotal.

## Status Surfaces

- `evonn-compare workspace-report <ws>` — refresh trends + dashboard.
- `evonn-compare evidence report --registry evidence` — compact decision
  surface.
- Per-engine `inspect` — runtime, usage, wins, leaders, failures from run
  artifacts.
- A root status helper SHOULD aggregate: current branch/commit per package,
  last promoted evidence, lane health, cache readiness.
