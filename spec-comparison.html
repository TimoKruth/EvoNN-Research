divergence 01

What “fair” means

claude-spec's rigor is comparative: budget contracts, contender floors, repeated seeds, lane states. claudex-spec's rigor is statistical: leakage, preprocessing scope, duplicate families, and contamination, enforced by process boundaries. They are complementary — one guards the comparison, the other guards the result.

divergence 02

The implicit code review

claudex-spec's “Fix” list is an audit of the current codebase that claude-spec inherits more trustingly:

  • unseeded MLX / NumPy / worker randomness
  • resume that doesn't reproduce uninterrupted runs
  • non-atomic checkpoints
  • parameter/byte proxies labeled as measurements
  • NEAT claims without active speciation
  • export paths that retrain

If accurate, these belong in either build.

divergence 03

Scope vs shippability

The risk profiles are opposite. claude-spec can be executed incrementally by one person but may re-encode today's debt. claudex-spec fixes the debt by construction but demands a web app, an orchestration layer, nine engines, and a security program before v1 — it may never reach done.

They compose: core and shell

Build the research core from claude-spec — engines, ladder, comparison evidence — and cherry-pick claudex-spec's contracts today: the evaluation artifact, the RNG stream model, atomic checkpoint publication, physical test-label separation, and the full “Fix” list. The claudex-spec product shell — dataset governance, orchestration, web UX, verified deployment bundles — can wrap that core later without changing what it proved.
