# 08 — Persistence, Artifacts, Provenance, and Retention

## 1. Database role separation

### Transactional metadata

The local profile uses SQLite in WAL mode with `synchronous=FULL`, accessed through SQLAlchemy 2 and one control-plane writer. PostgreSQL is an optional supported profile when remote access or higher write concurrency is required.

The metadata store owns projects, descriptors, plans, jobs, state transitions, tasks, attempts, workers, leases, policies, artifact metadata, model selections, exports, and audit events.

### Analytical evidence

Each job/run receives its own DuckDB database for bulk append/query of evaluations, metrics, resource samples, lineage, archives, and derived reporting tables.

DuckDB MUST NOT be the multi-process coordination database. While a run is active, exactly one owning evidence process holds the database and serves query results/snapshots to the control plane; separate dashboard/exporter processes MUST NOT open the file concurrently with the writer. At stable checkpoint/finalization boundaries, immutable database snapshots MAY be opened read-only by other processes.

### Artifact store

The local artifact store is content-addressed by a collision-resistant digest. It stores immutable blobs and manifests under a path independent of user labels.

## 2. Transactions and migrations

STORE-001: Metadata mutations MUST use explicit SQLAlchemy transaction boundaries.  
STORE-002: Recoverable item failures MAY use savepoints.  
STORE-003: Schema changes MUST use reviewed Alembic migrations.  
STORE-004: Autogenerated migrations are candidates only and MUST be reviewed, especially rename/drop operations.  
STORE-005: Upgrade and downgrade paths MUST be tested on fixtures and backups.  
STORE-006: Database schema version MUST be explicit and compatible ranges declared.

## 3. Canonical artifact descriptor

Every artifact MUST record:

- digest algorithm/value;
- byte size and media type;
- artifact type/schema version;
- producer job/attempt/component;
- creation time;
- logical role;
- dependencies and lineage;
- integrity and verification status;
- privacy/license classification;
- retention class;
- optional signatures/SBOM references.

Absolute local paths MUST NOT be exported as durable portable identity.

## 4. Publication protocol

1. Write to a private temporary path.
2. Stream-compute digest and size.
3. Validate schema/media/security rules.
4. Atomically move/link immutable content into the digest store.
5. In one metadata transaction, create the artifact reference, update the owning aggregate/checkpoint, and append an outbox event.
6. Publish the outbox event idempotently after commit.

A crash before step 4 leaves temporary garbage, not a published artifact. A crash after step 4 but before transaction commit leaves safe unreferenced content. A crash after commit but before event delivery is recovered from the transactional outbox. Metadata MUST never reference partially published content.

## 5. Native model bundle

The native bundle is dependency-complete rather than necessarily byte-self-contained. `dependency_mode: embedded` includes every required byte; `dependency_mode: content_addressed_local` binds any non-redistributable base assets to exact verified local digests. Both modes MUST verify fully offline and MUST fail if a required dependency is absent. The bundle is a manifest-driven directory/archive containing:

- bundle manifest;
- safe model weights/artifacts;
- model specification;
- preprocessing/postprocessing graph and fitted state;
- task/metric/schema bindings;
- input/output schemas;
- label/tokenizer/processor assets;
- runtime/backend requirements;
- model card and evidence summary;
- license/attribution notices;
- provenance and lineage;
- test vectors and expected outputs/tolerances;
- per-file checksums;
- optional platform-specific artifacts.

The bundle identity is the digest of the canonical manifest bytes plus their referenced file/dependency descriptors. The manifest does not contain its own digest; the outer artifact descriptor records it, avoiding self-reference.

## 6. Safe model formats

- Tensor weights SHOULD use safetensors where applicable.
- Scikit-learn-compatible objects SHOULD use skops or declarative reconstruction when supported.
- Pickle, cloudpickle, joblib, or arbitrary Python object deserialization MUST NOT occur for untrusted artifacts.
- If an unsafe format is unavoidable, the bundle MUST be marked trusted-local-only and require explicit approval in an isolated environment.
- MLX and PyTorch native artifacts MUST have exact framework/version requirements and safe-loading policies.

## 7. External model and dataset caches

Hugging Face assets MUST record repository, resolved commit hash, selected file paths, file hashes, license, processor/tokenizer revision, and local artifact digests. Mutable branch/tag names are convenience references only.

The standard cache MAY be used for deduplication but is not the experiment registry or authenticity proof.

## 8. OCI/ORAS

OCI/ORAS is optional for distribution, replication, signatures, SBOMs, and registry interoperability. Local execution MUST not require a registry.

OCI tags are mutable aliases; digests are immutable content identities. Published references in evidence MUST use digests.

## 9. Provenance graph

The provenance DAG MUST connect:

source artifact → verified dataset snapshot → split → preprocessing/fitted state → candidate spec → checkpoint/weights → evaluation → selection → finalization → export.

Unresolved required ancestors, cycles, fabricated timestamps, or invented success states invalidate decision-grade status.

## 10. Retention

Retention classes:

- temporary
- cache
- intermediate
- checkpoint
- final model
- promoted evidence
- legal hold

Promoted evidence and selected-model lineage MUST NOT be automatically deleted. Default cleanup SHOULD retain final manifests/results and at least the last two valid checkpoints while allowing quota-based removal of reproducible intermediates.

Deletion MUST honor shared references, governance, legal hold, and provenance. Legal hold takes precedence over ordinary retention/deletion until released by an authorized governance event. Privacy/license takedown creates a revocation/tombstone immediately, blocks further use/export, and identifies every derived dataset, evaluation, model, and bundle. Physical deletion then proceeds where legally required and permitted; immutable evidence metadata is retained only in redacted/tombstoned form sufficient to explain revocation, never as usable protected content. Derived models are deleted, quarantined, or marked unusable according to the governing policy and lineage. Garbage collection MUST be mark-and-sweep from durable roots and produce an audit report.

## 11. Evidence freshness

Claims about current engine/model status MUST cite evidence from the same code/registry set or a declared compatible range. Newer incompatible code automatically marks older role or recommendation evidence stale without deleting it.
